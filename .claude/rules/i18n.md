# i18n Rules

## Translation files

```
translations/
├── en.json      # source of truth — edit this first
├── de.json      # must mirror en.json structure exactly
└── en.d.json.ts # auto-generated types — never edit manually
```

**Always update both `en.json` and `de.json` in the same change.**
A key missing in `de.json` causes a runtime error in the German locale.

## Key structure

Keys are namespaced by top-level section:

```json
{
  "countries": {
    "page_title": "Countries of the World",
    "search_placeholder": "Search countries..."
  }
}
```

Naming: `snake_case`, descriptive, scoped to the feature (`countries.search_placeholder`
not `search.countries_placeholder`).

## Server Components

```typescript
import { getTranslations, setRequestLocale } from "next-intl/server";

// In page.tsx generateMetadata:
const t = await getTranslations({ locale, namespace: "page_titles" });
return { title: t("sign_in") };

// In page.tsx component body:
setRequestLocale(locale); // must be called before any rendering
```

`setRequestLocale` must be called in every page component before returning JSX.

## Client Components

```typescript
import { useTranslations } from "next-intl";

const t = useTranslations("countries");
// then use: t("page_title"), t("regions.africa"), etc.
```

- Namespace = top-level key in JSON (e.g. `"countries"`)
- Nested keys accessed with dot notation: `t("regions.africa")`
- Interpolation: `t("page_subtitle", { filtered: 10, total: 250 })`

## Navigation

```typescript
import { Link, useRouter, usePathname } from "@/pkg/locale";
// Never from "next/link" or "next/navigation" — those skip the locale prefix
```

## Adding a new translation key

1. Add to `translations/en.json` under the correct namespace
2. Add the same key to `translations/de.json` with the German value
3. The TypeScript type in `en.d.json.ts` updates automatically on next build
